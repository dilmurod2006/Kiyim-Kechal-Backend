# FastAPI E-Commerce Backend Architecture

This document provides a comprehensive overview of the `FastAPI-E-Commarce` backend project to help any AI or developer quickly understand its structure, technologies, and data flow.

## 🛠 Technology Stack
- **Web Framework:** FastAPI
- **Database:** PostgreSQL (Version 15)
- **ORM:** SQLModel (built on top of SQLAlchemy and Pydantic)
- **Database Drivers:** `asyncpg` (for async FastAPI endpoints), `psycopg2` (for synchronous Celery tasks)
- **Migrations:** Alembic
- **Background Tasks:** Celery
- **Message Broker & Task Backend:** Redis (Version 7)
- **Containerization:** Docker & Docker Compose

## 🗄 Database Setup
The project strictly uses **PostgreSQL** as its primary relational database. 
- The `compose.yaml` defines a `db` service running the `postgres:15` image.
- **Connection Strings** (from `core/config.py` and `compose.yaml`):
  - **Async URL:** `postgresql+asyncpg://postgres:postgres@db:5432/fastapi_ecom` (Used by FastAPI)
  - **Sync URL:** `postgresql+psycopg2://postgres:postgres@db:5432/fastapi_ecom` (Used by Celery workers)
- **Redis** is used as a supplementary in-memory data store, specifically functioning as the message broker and result backend for Celery background tasks.

## 📂 File Architecture
The project follows a modular, layered architecture for clear separation of concerns.

```text
FastAPI-E-Commarce/
├── alembic/                # Database migration scripts generated by Alembic.
├── alembic.ini             # Alembic configuration file.
├── api/                    # API Routers (Controllers). Handles HTTP requests and responses.
│   ├── background_status.py
│   ├── basic_background.py
│   ├── category.py
│   ├── product.py
│   ├── review.py
│   └── user.py
├── core/                   # Core application configurations and utilities.
│   ├── auth.py             # Authentication logic (JWT, etc.).
│   ├── config.py           # Environment variables and base settings (Pydantic BaseSettings).
│   ├── db.py               # Database async engine and session factory setup.
│   └── security.py         # Password hashing and security utilities.
├── crud/                   # Create, Read, Update, Delete (CRUD) operations. Data access layer.
│   ├── crud_category.py
│   ├── crud_product.py
│   ├── crud_review.py
│   └── crud_user.py
├── model/                  # Database Models & Schemas (SQLModel).
│   └── models.py           # Contains definitions for User, Category, Product, Review, ProductOrder.
├── schemas.py              # Pydantic schemas (if separated from SQLModel, or used for specific request/response validation).
├── main.py                 # Application entry point. Initializes FastAPI, configures CORS, and registers routers.
├── tasks.py                # Celery background tasks definitions (e.g., process_order).
├── worker.py               # Celery app initialization and broker/backend configuration.
├── compose.yaml            # Docker Compose configuration (defines web, db, redis, worker, migrate services).
├── Dockerfile              # Dockerfile for the web/worker/migrate containers.
└── requirement.txt         # Python dependencies.
```

## 🧩 Key Components

### 1. Database Models (`model/models.py`)
Defined using **SQLModel** (which doubles as both SQLAlchemy models and Pydantic validation schemas):
- **User**: Represents customers/admins. Has a relationship with `Review`.
- **Category**: Product categories. Has a relationship with `Product`.
- **Product**: E-commerce items. Linked to `Category` and `Review`.
- **Review**: Customer reviews for products. Linked to `User` and `Product`.
- **ProductOrder**: Tracks order status (`Packaging`, `Shipping`, `Delivered`).

### 2. API Layer (`api/`)
The routers are modularly imported into `main.py`. Each file in `api/` focuses on a specific domain (e.g., `user.py` for user endpoints) and delegates database interactions to the `crud/` layer to keep the routing logic clean.

### 3. Background Processing (`tasks.py` & `worker.py`)
- Uses **Celery** to process long-running tasks asynchronously to avoid blocking the FastAPI event loop.
- Example: The `process_order` task simulates a multi-stage order fulfillment process, updating the `ProductOrder` status in PostgreSQL synchronously via `psycopg2` while the FastAPI app runs asynchronously.
- Dependent on the `redis` service in Docker.

## 🚀 How to Run (Docker)
The environment is fully containerized. Running `docker compose up` spins up:
1. `db`: PostgreSQL server
2. `redis`: Redis server
3. `migrate`: Automatically runs Alembic migrations (`alembic upgrade head`)
4. `worker`: Celery worker to consume background tasks
5. `web`: Uvicorn server running the FastAPI app on port 8000
